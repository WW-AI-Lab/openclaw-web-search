# openclaw-web-search 开发最佳实践

本文档描述如何把 `openclaw-web-search` 作为一个独立 Git 仓库、独立 npm 包来开发，
并通过 OpenClaw 的插件安装机制加载，而不是继续把实现放在 OpenClaw 主仓库里。

## 目标定位

这个目录的目标不是主仓库里的 `extensions/*` 风格内置插件，而是一个独立外部插件仓库：

- 独立 Git 历史
- 独立 npm 包
- 独立版本发布
- 通过 `openclaw plugins install` 或 `openclaw plugins install -l` 安装

这意味着后续真实实现应尽量只依赖公开 SDK：

- `openclaw/plugin-sdk/plugin-entry`
- `openclaw/plugin-sdk/provider-web-search`

不要依赖 OpenClaw 主仓库里的 `src/**` 私有实现路径。

## 目录结构建议

当前目录已经按独立插件仓库骨架初始化：

```text
openclaw-web-search/
  .git/
  .gitignore
  package.json
  tsconfig.json
  openclaw.plugin.json
  index.ts
  src/
    provider.ts
  docs/
    开发最佳实践.md
```

后续建议继续保持这个边界：

- `index.ts` 只负责注册
- `src/provider.ts` 负责 `WebSearchProviderPlugin`
- 如果后面接入真实搜索后端，再拆分：
  - `src/client.ts`
  - `src/types.ts`
  - `src/provider.test.ts`

## 包与插件标识

当前骨架统一使用同一组标识：

- 目录名：`openclaw-web-search`
- npm 包名：`openclaw-web-search`
- 插件 id：`openclaw-web-search`

这样做的好处是：

- 安装说明简单
- 用户配置路径稳定
- 减少包名和插件 id 不一致带来的运维成本

除非后续需要 scope 包名，否则不要轻易拆分成不同名字。

## 为什么要独立仓库

把 `web_search` 放在独立仓库有几个现实好处：

- 可以独立发版，不必跟随 OpenClaw 主仓库发布周期
- 可以单独管理搜索服务的实验性变更
- 可以在私有环境里开发企业内部搜索接入
- 出现 API 波动时，回滚和热修更简单

对 `web_search` 这类外部服务集成来说，这种边界通常比主仓库内开发更稳。

## 实现边界

一个合格的外部 `web_search` 插件，应该自己负责：

- 凭据读取
- 请求参数校验
- 请求发送
- 响应归一化
- 摘要与引用链接生成
- 缓存 key 设计

不要把这些逻辑散落到多个 OpenClaw 宿主层脚本里。

宿主 OpenClaw 负责的是：

- 加载插件
- 管理启用状态
- 通过插件系统把 `web_search` 能力接入 agent

## 配置设计建议

当前 `openclaw.plugin.json` 采用最小配置面：

- `webSearch.apiKey`
- `webSearch.baseUrl`
- `webSearch.timeoutSeconds`

这是一个合理起点。后续新增配置时，遵守这几个原则：

1. 先保守，后扩展。
2. 不要把上游 API 的每个参数都直接暴露给用户。
3. 只暴露真正稳定、重复使用、能改善体验的参数。
4. 凭据优先支持配置值和环境变量双路径。

不建议第一版就暴露：

- 十几个 provider 专属开关
- 多层嵌套搜索策略
- 大量调试字段

## 凭据最佳实践

优先顺序建议如下：

1. `plugins.entries.openclaw-web-search.config.webSearch.apiKey`
2. `OPENCLAW_WEB_SEARCH_API_KEY`

这样有几个好处：

- 本地开发简单
- 部署时便于注入环境变量
- 用户可以逐步从环境变量迁移到配置或 secret ref

另外两条规则要坚持：

- 不要把 API key 拼进缓存 key
- 不要把敏感值写进错误信息

## 网络与安全边界

对外部 `web_search` 插件，最重要的不是“能请求成功”，而是“请求边界可控”。

所以实现时应优先使用 SDK 提供的受控 helper：

- `withTrustedWebSearchEndpoint(...)`
- `wrapWebContent(...)`

这样做的原因：

- 统一超时与受信请求边界
- 保持外部网页内容与本地指令的信任隔离
- 避免未来宿主升级后，插件侧安全行为不一致

如果以后要支持站点过滤、时间过滤、语言过滤，也应继续走受控 helper，而不是自行绕开。

## 返回结果设计

`execute(...)` 的返回结构应保持稳定、克制、可审计。

建议长期保持这些字段：

- `query`
- `provider`
- `content`
- `citations`

可选字段：

- `tookMs`
- `rawCount`
- `mode`

不建议默认返回：

- 上游 API 原始响应全文
- 调试 token
- 大块未清洗 HTML

原则很简单：给 agent 足够用的信息，不要把宿主暴露给不稳定原始 payload。

## 缓存策略

`web_search` 很适合缓存，但缓存设计要保守。

推荐纳入 cache key 的字段：

- provider id
- query
- count
- baseUrl
- 会改变结果语义的过滤条件

不要纳入的字段：

- API key
- 临时 trace id
- 无语义影响的调试参数

第一版建议先使用 SDK 现成缓存 helper，不要自建第二套磁盘缓存。

## 本地联调方式

这个目录未来应按“外部插件”方式联调，而不是复制到 `extensions/`。

推荐流程：

```bash
cd /path/to/openclaw-web-search
npm install

cd /path/to/openclaw
openclaw plugins install -l /path/to/openclaw-web-search
openclaw plugins enable openclaw-web-search
openclaw gateway restart
```

如果不想走 `--link`，也可以手工写宿主配置：

```json5
{
  plugins: {
    load: {
      paths: ["/path/to/openclaw-web-search"]
    },
    entries: {
      "openclaw-web-search": {
        enabled: true
      }
    }
  }
}
```

关键点：

- 外部插件目录要能被 OpenClaw 发现
- 插件 id 必须与 manifest 一致
- 启用后需要重启 gateway

## 测试建议

至少应补上这些测试：

- 缺少凭据时报错正确
- `query` / `count` 参数校验正确
- 请求 body 序列化正确
- 响应被正确归一化为 `content` + `citations`
- 缓存命中路径正确
- 非 2xx 响应能输出可诊断错误

推荐分层：

- `client` 层做 HTTP 级单测
- `provider` 层做工具输出单测
- 真实线上 API 只保留可选 smoke test

不要把线上联网测试当成默认 CI 主路径。

## 发布建议

后续真正发布时，建议这样做：

- 该目录在独立仓库中维护自己的 tag 和 changelog
- npm 包独立发版
- `openclaw` 作为 `peerDependencies`
- README 里明确写最低支持的 OpenClaw 版本

如果是私有插件，也可以：

- 发布到私有 npm registry
- 或者打包为 `.tgz` 后通过 `openclaw plugins install ./xxx.tgz` 安装

## 开发演进路线

建议按下面顺序推进，而不是一次性做满：

1. 先把当前骨架跑通
2. 接入一个真实搜索后端
3. 让返回结果包含摘要和引用
4. 加入缓存和超时控制
5. 再考虑 provider 专属高级参数
6. 如果确有必要，再增加配套工具，例如 extract 或 scrape

这条路线能最大限度减少早期复杂度。

当前目录已经具备后续开发的基础前提：

- 是独立 Git 仓库
- 是独立 npm 包
- 有 OpenClaw 原生插件 manifest
- 有 `registerWebSearchProvider(...)` 最小入口
- 有中文开发文档
